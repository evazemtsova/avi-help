## 1. TL;DR

Делаем AI-поиск по справочному центру support.avito.ru: пользователь задаёт вопрос человеческим языком — получает короткий обзорный ответ от LLM со ссылками на исходные статьи (как AI Overview в Google). Альтернатива текущему сайту, где есть только полнотекстовый поиск с выдачей в виде списка ссылок.

Прототип UI уже есть (html с моками). Задача — заменить моки на реальный RAG, развернуть на Vercel (фронт) + Railway (API + индекс), собрать минимальный eval.

## 2. Зачем (контекст и мотивация)

### 2.1. Бизнес-проблема (формулирую как если бы это была реальная команда)

Авито саппорт получает ~N млн обращений в год. Часть из них — вопросы, на которые ответ уже есть в базе знаний (`support.avito.ru`), но пользователь либо не нашёл, либо не понял длинную статью. Это создаёт нагрузку на операторов и снижает CSAT (пользователь раздражён до того, как написал).

Сценарий «сам разобрался по справке» — самый дешёвый канал. Любой процент вопросов, который мы переводим из «написал в чат» в «прочитал AI-ответ и ушёл», экономит деньги и улучшает опыт.

### 2.2. Личная мотивация (контекст пет-проекта)

Откликаюсь на вакансию DS Product Manager в команду автоматизации саппорта. Нет коммерческого ML-опыта → нужно показать на конкретном артефакте, что:
- понимаю, как устроен RAG end-to-end (от парсинга до eval);
- умею думать продуктово: метрики, гипотезы, баланс цена/ценность;
- могу довести идею до работающего демо за реалистичный срок.

Проект максимально близок к их домену — это не абстрактный «чатбот по вики», а решение их же задачи на их же контенте.

## 4. Аудитория и сценарии использования

### 4.1. Целевые пользователи (имитируем как для прод-фичи)

- **Покупатель** (60% запросов в саппорт по моей оценке): «как вернуть деньги», «продавец не отвечает», «как работает безопасная сделка».
- **Продавец** (30%): «как разместить объявление», «почему отклонили», «как поднять в поиске».
- **Жертва мошенничества** (5%): «звонят с поддельной ссылкой», «увели деньги».
- **Прочее** (5%): доставка, путешествия, работа.

### 4.2. Топ user stories

1. **«Быстрый ответ на простой вопрос»** — пользователь печатает вопрос, видит обзор за 5 секунд, понимает что делать, не идёт в чат.
2. **«Не нашёл в обзоре — проваливается в источник»** — обзор недостаточный, пользователь кликает на пилюлю источника, открывается оригинальная статья на support.avito.ru.
3. **«Ответ не помог — оценил негативно»** — пользователь жмёт thumbs down, мы это логируем для eval.

### 4.3. Что НЕ делает пользователь в MVP

- Не задаёт уточняющие вопросы (multi-turn).
- Не логинится.
- Не пишет в чат с оператора прямо из интерфейса (только ссылка наружу).

## 4.4. Источник данных

- **Объём:** **518 статей** в справочном центре. Источник — внутренний JSON API `support.avito.ru/api/1/getCatalog`, возвращающий иерархическую структуру каталога: 16 категорий (`typeId=1`) + 70 разделов (`typeId=2`) + 518 статей (`typeId=4`).
- **Получение контента:** `POST /api/1/article` с `{"id": <id>}` возвращает JSON со всеми полями статьи, включая `title`, `body` (HTML-фрагмент с тегами `<headline>`, `<spoiler>`, `<tabset>`, `<factoid>`), `sectionId`, `categoryId`, `alias`.
- **Формат URL:** `/articles/{id}` — пример: `/articles/1833`. У одной статьи из 518 нет URL — обрабатываем как edge case (логируем, пропускаем).
- **Доступ:** API внутренний, не задокументирован публично, но работает без авторизации и rate limit-ов в разумных пределах (проверили 40 параллельных запросов с конкуррентностью 8 — ни одного 4xx/5xx). Фронт сайта блокирует прямой fetch с серверов вне РФ → парсим с локальной машины, артефакт коммитим в репо как `data/articles.jsonl`.
- **Метаданные из API:** `categoryId`, `sectionId` (с возможностью резолва в человекочитаемые названия через каталог), `alias`, `isActive`. Это означает, что иерархия «категория / раздел / статья» доступна напрямую — не надо скрейпить навигацию.
- **Время полного парсинга:** 1–2 минуты при `Semaphore(8)` (~25 мс на запрос).

## 4.5. Решение про классификатор интента (явный отказ в MVP)

**Не делаем отдельный классификатор интента в MVP.** Обоснование:

- На 518 статьях с понятной таксономией embeddings + retrieval уже хорошо разводят запросы по семантике: «у меня списали деньги» → чанки про мошенничество, «как разместить» → инструкция продавцу.
- Zero-shot классификация через LLM = ещё один вызов LLM на каждом запросе → плюс 1–2 сек latency и плюс к стоимости, без явного выигрыша против baseline.
- Обученный классификатор требует размеченных данных, которых нет.

**Что делаем вместо:** категория из хлебных крошек храним как метаданные у каждого чанка в векторной БД. Используем её в двух местах:
1. **UI:** реальная категория Авито в пилюле источника (не выдуманные «М»/«О»/«Б» из прототипа).
2. **Промпт-усиление:** если все top-3 чанка retrieval из категории «Безопасность» → системный промпт добавляет жёсткое предупреждение про коды из SMS и ссылки. Это постфильтр по найденному, а не классификация запроса.

**Когда вернёмся к идее (раздел 12, roadmap):** после MVP, на собранных логах запросов и фидбека (👍/👎). Тогда будет данные для разметки и понимание, на каких интентах baseline проседает.

## 5. Скоуп (что войдёт в 5 дней)

### 5.1. MUST (без этого нет проекта)

- **M1.** Парсер через JSON API — забирает 518 статей через `POST /api/1/article`, парсит HTML-фрагмент из поля `body`, извлекает структурные блоки (`<headline>`, `<spoiler>`, `<tabset>`, `<factoid>`); режет на чанки по структурной стратегии (детали в ML System Design).
- **M2.** Индексация — embeddings (OpenAI text-embedding-3-small) + векторная БД (Qdrant либо Chroma локально на диске Railway). У каждого чанка в payload: `article_id`, `article_url`, `title`, `category`, `lastmod`, `chunk_text`.
- **M3.** Retrieval-ручка `/search` — принимает запрос, возвращает топ-K релевантных чанков.
- **M4.** Generation-ручка `/answer` — принимает запрос, делает retrieval + генерит ответ через Claude (Haiku по дефолту, опция Sonnet) с цитированием источников.
- **M5.** Интеграция с существующим html-прототипом — заменяем `mockAnswers` на fetch к API.
- **M6.** Деплой: фронт на Vercel, API + индекс на Railway.
- **M7.** Eval: 30–50 вопросов с эталонными источниками + автоматический скрипт.
- **M8.** Страница `/about` или README с описанием архитектуры, метриками, стоимостью.

### 5.2. SHOULD (если останется время)

- **S1.** Стриминг ответа (SSE) — буквы появляются по мере генерации, перцептивно быстрее.
- **S2.** Логирование запросов и фидбека (thumbs up/down) в Postgres на Railway.
- **S3.** Reranker — `bge-reranker` или второй проход через LLM по топ-20 для топ-3.
- **S4.** Кэш ответов на популярные запросы (Redis на Railway либо просто in-memory dict).

### 5.3. WON'T (явно за рамками)

- Многоязычность.
- Защита от prompt injection (упомянем в TDR как риск, но не реализуем).
- Rate limiting (поставим базовый на уровне Railway, не в коде).
- Полноценный мониторинг (Grafana/Prometheus). Используем дефолтный Railway dashboard.

## 6. Функциональные требования

### 6.1. Поиск с AI Overview

- **F1.** Поле ввода ≤200 символов, единственное обязательное поле.
- **F2.** При сабмите — спиннер «Обзор от ИИ генерируется…» (уже в прототипе).
- **F3.** Ответ показывается в карточке: lead-параграф (1–2 предложения суть) + опциональные секции списком + блок источников.
- **F4.** Источник — пилюля с буквой и названием категории (как в прототипе), кликабельна, ведёт на оригинальную статью на support.avito.ru.
- **F5.** Если ответ длиннее ~10 строк — кнопка «Развернуть» (уже в прототипе).
- **F6.** Кнопки фидбека: 👍 / 👎 / поделиться. Лайки логируются.
- **F7.** Если retrieval не нашёл релевантного (низкие скоры) — показываем fallback: «По запросу ничего точного не нашлось. Можно [написать в поддержку]».

### 6.2. Качество и безопасность ответа

- **F8.** Ответ всегда содержит ≥1 источник из БЗ. Без источника — не отвечаем (показываем fallback из F7).
- **F9.** LLM должна отказываться отвечать на вопросы вне домена Avito (политика, новости, медицина и т.д.) — через системный промпт.
- **F10.** Не выдумывать факты. Если в чанках нет ответа — fallback из F7. (Защита от галлюцинаций — в системном промпте + темплейт ответа.)

### 6.3. Latency и надёжность (целевые метрики)

- **F11.** P50 latency end-to-end ≤4 сек, P95 ≤8 сек (на Haiku). На Sonnet — P50 ≤7 сек, P95 ≤12 сек.
- **F12.** Аптайм демо в день презентации — best effort. Падение Railway не бьёт по фронту фатально (показываем читаемую ошибку).

## 7. Метрики и как меряем

### 7.1. Продуктовые (для имитации работы PM)

| Метрика | Как считаем | Целевое значение в демо |
|---|---|---|
| Resolution rate (прокси) | % сессий, где пользователь не нажал «написать в поддержку» после ответа | На демо смотрим как baseline, не таргетируем |
| Helpful rate | 👍 / (👍 + 👎) | Сам себе ставлю на 50 ручных запросах: ≥70% |
| Avg time to answer | время от сабмита до полного ответа | P50 ≤4 сек |
| Coverage | доля запросов, где retrieval вернул хотя бы 1 чанк выше порога score | ≥85% на eval-сете |

### 7.2. ML-метрики (eval-pipeline)

Делаем golden-set из 30–50 вопросов. Для каждого — рукой проставляю эталонные `article_url`, которые ДОЛЖНЫ быть в источниках.

| Метрика | Что меряем | Цель |
|---|---|---|
| Recall@5 (retrieval) | % вопросов, где эталонный URL попал в top-5 retrieval | ≥0.85 |
| MRR@10 (retrieval) | mean reciprocal rank | ≥0.6 |
| Faithfulness (generation) | % ответов, где утверждения подтверждаются цитируемыми источниками. Считаем через LLM-as-judge (Claude Sonnet оценивает ответ Haiku). | ≥0.9 |
| Answer relevance | насколько ответ отвечает на вопрос (LLM-as-judge, 1–5) | avg ≥4 |
| Refusal rate on out-of-domain | % корректных отказов на 10 заранее заготовленных вне-доменных вопросов | =1.0 |

### 7.3. Экономика

**Разовая индексация ~518 статей** (text-embedding-3-small, $0.02/1M tokens):
- Оценка: ~5200 чанков × ~250 токенов = ~1.3M токенов → **~$0.025** на полный реиндекс.
- Можно гонять переиндексацию хоть каждый день, бюджет не пострадает.

**Стоимость 1 запроса** (retrieval уже бесплатный из локальной БД, считаем embedding запроса + LLM-генерация):
- **Cheap:** Haiku + text-embedding-3-small. ~$0.001–0.002/запрос.
- **Quality:** Sonnet + text-embedding-3-small + reranker. ~$0.01–0.02/запрос.

**Бюджет на 5 дней разработки:**
- Индексация: ~$0.05 (с переиндексами).
- Eval-прогоны: 50 вопросов × ~5 итераций × Haiku = ~$0.5.
- Демо-запросы и ручная проверка: ~$2.
- **Итого: ~$3** из имеющихся $15. Запас на эксперименты (Sonnet-варианты, judge-модель в eval) сохраняется.

**Экстраполяция на масштаб** (для собеса):
- 1М запросов/мес на Haiku ≈ $1–2k/мес — сравним с зарплатой одного оператора.
- 1М на Sonnet ≈ $10–20k/мес — окупается, если каждый ответ снимает ≥0.05 минуты оператора.

## 9. Риски и допущения

### 9.1. Технические риски

| Риск | Вероятность | Митигация |
|---|---|---|
| Авито меняет JSON API (без объявления — он внутренний) | низкая на горизонте 5 дней | Парсим однократно с локальной машины. Артефакт `articles.jsonl` в репо. Если API сломается ПОСЛЕ парсинга — индекс уже собран, ничего не блокируется. Если сломается ДО — переписываем парсер на playwright за ~2 часа (HTML-структура та же). |
| Бюджет API кончается на eval | низкая | Прогон 50 вопросов на Haiku ≈ $0.10. Кэшируем все вызовы LLM в файл — повторный прогон бесплатный. |
| Railway не вывозит индекс + API | очень низкая | Chroma на диске. ~5200 чанков × 1536 dim = ~32 MB embeddings + ~10 MB метаданные ≈ 42 MB. На 8GB-плане Railway это <1% памяти. |
| LLM галлюцинирует на отказных кейсах | высокая | Системный промпт + проверка «есть ли источник в ответе» на этапе постпроцессинга. |
| 5 дней не хватит | средняя | См. план в TDR — режем с конца, MUST-фичи первыми. |

### 9.2. Допущения

- support.avito.ru предоставляет внутренний JSON API без авторизации (проверено: 40 параллельных запросов прошли успешно, без 4xx/5xx). Если API закроют до момента парсинга — есть playwright-фолбэк за ~2 часа.
- Контент справки — это структурированный HTML (`<headline>`, `<spoiler>`, `<tabset>`, `<factoid>`, таблицы), без интерактивных элементов.
- Я могу за день написать парсер + индекс + базовую ручку (это honest assessment — раньше делал похожее).

- Multi-turn (пользователь уточняет вопрос).
- **Классификатор интента** на собранных логах + фидбеке. Вернёмся к идее, когда будут данные для разметки. Пилотные классы: «мошенничество», «деньги/возврат», «как разместить», «модерация», «доставка», «прочее». Зачем нужен: разные SLA, разные темплейты ответа, разные эскалации (на «мошенничестве» сразу подтягиваем спецоператора).

**Через 1–2 месяца:**
- Дообучение reranker на собранном фидбеке (👍/👎).
- Классификатор интента → разные промпты для покупателей / продавцов / жертв мошенничества.
- Интеграция с системой тикетов: если AI overview не помог → preform тикета с уже собранным контекстом.
- Self-serve расширение БЗ — операторы саппорта помечают, какие новые вопросы плохо отвечаются.

**Через 3–6 месяцев:**
- Свой sentence-encoder, дообученный на парах (вопрос пользователя → статья БЗ).
- LLM-judge как отдельный сервис для непрерывного eval.
- Фоновая дедупликация и обновление БЗ при изменении статей на support.avito.ru.
